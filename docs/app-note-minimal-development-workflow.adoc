= app-note 最简开发流程规范（V1）
:toc: left
:toclevels: 4
:sectnums:

== 文档定位

- 目标：在“开发人数少、经验有限、AI 高参与”的前提下，定义一套可执行、可复用、低维护成本的最简开发流程。
- 适用范围：`app-note` 全仓库（后端 `boot/app-layer/data-core/module/*`、前端 `note-ui`、接口调试与脚本目录）。
- 目标角色：一个初级程序员 + AI 协作。
- 当前版本原则：只落地你确认的 7 步主流程，不引入额外治理负担。

== 全局约束（必须遵守）

=== 约束 1：先文档，后实现

任何功能开发前，必须先补齐本流程要求的最小文档产物；文档未完成，不进入编码。

=== 约束 2：先原型，后接口

前端页面和交互先以静态 HTML 原型确定；原型未评审通过，不进入后端模型设计。

=== 约束 3：一次只做一件事

一个开发任务只对应一个业务目标，避免“顺手改动”导致回归风险扩散。

=== 约束 4：改动必须可验证

任何代码改动必须附带可执行验证动作（命令或脚本）；不能验证的改动不允许合入主分支。

== 7 步最简开发流程

== 第一步：定义开发规范与风格

目标：

- 统一 AI 与人工输出风格，避免页面、接口、代码风格漂移。

最小输入：

- 功能需求简述（1~2 段）
- 当前模块归属（例如 `module/server-practice` 或 `note-ui`）

最小输出（必须产物）：

- `docs/standards/development-baseline.adoc`（统一规范基线）

本步必须定义的内容：

- 命名规范（接口、DTO、VO、实体、前端组件）
- 分层边界（Controller / Service / Repository / Entity）
- 页面风格基线（颜色、间距、字体、组件风格）
- AI 生成约束（禁止越层调用、禁止跨域写逻辑）
- 提交规范（分支、提交信息、是否允许直接推主分支）

完成标准：

- 新任务开始时，AI 和开发都以同一份规范为唯一基准。

== 第二步：创建高保真静态产品页面（HTML）

目标：

- 产出可连续交互的产品页面，用于确认需求与实现边界。

=== 2.1 创建静态产品页面

最小输入：

- 功能页面清单
- 页面间跳转关系

最小输出（必须产物）：

- `static-product/index.html`

执行规则：

- 原型必须是可在浏览器直接打开的静态 HTML。
- 页面要做到“接近真实产品”：布局、交互流程、状态切换应与正式页面基本一致。
- 页面之间必须可连续跳转和操作，不能只做孤立页面截图式原型。
- 仅数据来源静态化：数据写死在页面或本地静态 JSON，不从后端获取数据。
- 允许按模块增加子目录（例如 `static-product/practice/`、`static-product/word-card/`），但主入口保持 `static-product/index.html`。
- UI 评审以页面为准，不以口头描述为准。

=== 2.2 完善业务产品说明文档

最小输入：

- 已完成的静态产品页面（2.1）

最小输出（必须产物）：

- `docs/业务产品文档.adoc`

执行规则：

- 必须逐页补齐“页面功能与边界”。
- 必须标注页面开发状态（新建/开发中/测试中/开发完成）。
- 页面开发状态用于任务推进与 AI 去重，避免重复执行已完成页面需求。

完成标准：

- 页面可交互结果与文档定义一致，且每个页面职责和边界可追溯。

== 第三步：定义后端架构与核心数据结构

目标：

- 明确业务实体关系、接口边界、模块归属，确保后续可实现且可维护。

最小输入：

- 以下二选一即可进入：
- 第二步确认后的某个静态产品页面（`static-product/...`）
- `docs/业务产品文档.adoc` 中对应页面条目（优先选择状态为“新建/开发中/测试中”的页面）
- 现有模块架构文档（例如 `docs/app-note-architecture.adoc`）

最小输出（必须产物）：

- `docs/architecture/<feature-name>-backend-architecture.adoc`

本步必须包含：

- 业务流程说明（按请求链路）
- 模块落位（放在哪个模块，为什么）
- 核心表/实体设计（新增、修改、约束）
- API 列表（路径、方法、入参、出参）
- 与现有领域（`data-core`、`server-word`、`server-practice`）的关系

完成标准：

- 开发人员可以只看该文档完成后端编码，不需要二次口头澄清。

== 第四步：前后端实现（允许补充详细设计）

目标：

- 按文档将能力落地到代码，保证实现与设计一致。

最小输入：

- 第三步后端设计文档
- 第二步静态产品页面

最小输出（必须产物）：

- 后端实现代码（对应模块）
- 前端实现代码（`note-ui`）
- 可选：`docs/detail/<feature-name>-frontend-detail.adoc`（复杂页面时补）

执行规则：

- 后端：遵守模块边界，不跨层写业务。
- 前端：页面结构以原型为准，交互行为按后端 API 实现。
- AI 生成代码后，必须人工做最小检查：命名、分层、异常处理、空值处理。

完成标准：

- 本地可运行，关键路径可手动操作闭环。

== 第五步：测试（自动化 + 人工复核）

目标：

- 把“以为只影响局部”的风险变为“可见的验证结果”。

最小输入：

- 本次改动清单
- 影响接口清单

最小输出（必须产物）：

- `docs/test-reports/<feature-name>-test-report.adoc`

本步必须执行：

- 后端自动化测试：`mvn test`
- 前端基础检查：`cd note-ui && npm run lint`
- 接口回归（可用 `http/*.http` 或脚本）
- 人工复核核心链路（至少 1 次完整操作）

测试报告最少内容：

- 执行时间
- 执行命令
- 通过/失败结果
- 失败项与处理结论
- 人工复核结论

完成标准：

- 自动化结果可追溯，人工确认结果可追溯。

== 第六步：部署（脚本化）

目标：

- 让部署过程可重复、可交接、可稳定执行。

最小输入：

- 第五步测试通过结果
- 目标环境信息（测试/正式）

最小输出（必须产物）：

- `scripts/deploy-<env>.sh`（如不存在则新增）
- `docs/deploy/deploy-<env>.adoc`

部署文档至少包含：

- 部署前检查项
- 部署命令
- 配置项说明
- 部署后验证命令

完成标准：

- 非作者按文档和脚本可以完成同样部署。

== 第七步：验收

目标：

- 验证发布结果与需求目标一致，且核心功能可用。

最小输入：

- 部署完成记录
- 需求目标清单

最小输出（必须产物）：

- `docs/acceptance/<feature-name>-acceptance.adoc`

验收清单（最小）：

- 核心功能可用
- 关键接口可用
- 页面关键路径可用
- 历史关键功能未被破坏（最小回归）

完成标准：

- 验收文档明确“通过/不通过”及问题清单。

== 任务执行模板（给初级程序员）

=== 单任务标准目录

[source,text]
----
docs/
|- standards/
|  `- development-baseline.adoc
|- architecture/
|  `- <feature-name>-backend-architecture.adoc
|- detail/
|  `- <feature-name>-frontend-detail.adoc  (可选，详细设计)
|- test-reports/
|  `- <feature-name>-test-report.adoc
|- deploy/
|  `- deploy-<env>.adoc
`- acceptance/
   `- <feature-name>-acceptance.adoc

static-product/
|- index.html
`- <module-name>/  (可选，按模块扩展)
----

=== 单任务执行顺序（必须按顺序）

1. 建立/更新规范基线
2. 完成第二步（2.1 静态产品页面 + 2.2 业务产品说明文档）
3. 完成后端架构设计文档
4. 开发前后端代码
5. 执行自动化测试 + 人工复核
6. 执行脚本化部署
7. 完成验收记录

== 当前阶段暂不实现，但后续必须补齐

说明：以下内容本版先不落地，避免当前维护成本上升；但它们对系统长期稳定性非常关键。

=== A. 接口契约冻结（建议优先级：高）

重要性：

- 在前后端并行开发时，接口频繁变更是返工主因。
- 先冻结 API 契约可显著减少联调成本。

后续落地建议：

- 在第三步输出中新增“API 契约清单”并设变更流程。

=== B. 可回滚机制（建议优先级：高）

重要性：

- 没有回滚脚本时，一次上线问题会直接影响生产可用性。

后续落地建议：

- 第六步补充 `rollback-<env>.sh` 和回滚演练记录。

=== C. 基础监控与告警（建议优先级：中高）

重要性：

- 只做功能验收无法及时发现线上异常。
- 监控是“上线后可持续维护”的前提。

后续落地建议：

- 增加关键接口成功率、错误率、响应时间监控。

=== D. 统一接口自动回归流水线（建议优先级：中）

重要性：

- 纯手动回归不稳定，且人力成本随功能增长快速上升。

后续落地建议：

- 将 `http/*.http` 与测试命令纳入 CI 自动执行。

== 版本说明

- V1（当前）：只落地 7 步主流程，控制治理成本。
- V2（后续）：逐步引入契约冻结、回滚与监控，不一次性加重团队负担。
