= 前端开发规范-目录架构设计
:toc: left
:toclevels: 3
:sectnums:

== 1. 目的与适用范围

本文档用于统一 Vue2 前端项目的目录架构与职责边界，解决以下问题：

* 页面代码难定位
* 业务目录混用、复用边界模糊
* 测试页面进入生产环境风险
* 数据模型定义散乱，维护成本高

适用范围：本仓库所有 Vue2 前端业务开发。

== 2. 顶层目录规范

[source,text]
----
src/
  api/                 # 后端接口管理（url、参数、请求封装）
  assets/              # 静态资源
  components/          # 架构级通用组件（跨项目可复用）
  router/              # 路由管理
  views/               # 页面管理（按首页/业务/业务通用/测试分层）
  model/               # 数据模型管理（后端模型 + 前端模型）
----

=== 2.1 api

* 仅负责接口请求能力，不承载页面状态。
* 按业务模块拆分文件，如 `api/auth.js`、`api/order.js`。

=== 2.2 assets

* 存放图片、字体、样式资源等静态文件。
* 目录名统一使用 `assets`（避免 `asserts` 拼写混用）。

=== 2.3 components

* 存放架构级通用组件。
* 要求无业务语义，可在新项目直接复用。
* 示例：通用弹窗、通用表格容器、通用分页容器。

=== 2.4 router

* 统一路由入口与模块路由注册。
* 测试路由独立管理，默认不进入生产构建。

=== 2.5 views

* 仅负责页面与页面级业务模块组织。
* 目录结构见第 3 章。

=== 2.6 model

* 统一管理数据模型定义。
* 分为后端接口模型与前端状态模型，见第 4 章。

== 3. views 目录规范

[source,text]
----
views/
  home/                # 首页及与首页有关页面
  domain/              # 业务页面（按业务域拆分）
    auth/
    order/
  commons/             # 业务通用组件（有业务语义，跨业务复用）
  test/                # 调试/测试页面（默认不进生产路由）
----

=== 3.1 home

* 存放首页及首页相关页面。
* 首页相关逻辑不得散落到其他业务域目录。

=== 3.2 domain

* 所有业务页面集中在 `views/domain` 下。
* 业务域按语义命名，例如 `auth`、`order`、`customer`。
* 单一业务域下可继续按页面或子模块拆分，避免单目录文件过多。

=== 3.3 commons

* 存放业务通用组件。
* 该类组件可带业务语义，但需具备跨多个业务域复用价值。
* 禁止将 `views/commons` 作为“新业务目录”使用。

=== 3.4 test

* 存放联调、调试、演示页面。
* 必须通过独立测试路由注册。
* 生产环境默认禁用访问。

== 4. model 数据模型规范

== 4.1 设计原则

数据模型分两类：

* 后端相关模型：请求/响应结构定义
* 前端相关模型：页面与交互状态定义

=== 4.2 后端模型（跟 api 目录走）

[source,text]
----
model/
  api/
    auth/
      login.request.ts
      login.response.ts
      refresh-token.request.ts
      refresh-token.response.ts
    order/
      query-list.request.ts
      query-list.response.ts
----

约束：

* 与 `api/` 按业务域同构组织。
* 一个接口动作对应 `*.request.ts` 与 `*.response.ts`。
* 文件命名必须体现动作语义，禁止 `data.ts`、`model.ts` 之类泛命名。

=== 4.3 前端模型（跟 views 目录走）

[source,text]
----
model/
  views/
    home/
      home.state.ts
    domain/
      auth/
        login.state.ts
      order/
        order-list.state.ts
        order-detail.state.ts
----

约束：

* 与 `views/` 目录层次对齐，保证可快速定位。
* 仅定义前端自行维护状态，如：筛选条件、表单态、UI态、分页态。

=== 4.4 依赖方向约束

* `api/**` 可依赖 `model/api/**`。
* `views/**` 可依赖 `model/api/**` 与 `model/views/**`。
* `model/api/**` 不得依赖 `views/**`。
* `model/views/**` 不得依赖 Vue 页面组件实现。

== 5. router 路由组织规范

建议拆分：

* `router/base.js`：正式业务路由（home + domain）
* `router/test.js`：测试路由（views/test）
* `router/index.js`：统一聚合并按环境控制

示例：

[source,javascript]
----
import Vue from 'vue'
import Router from 'vue-router'
import baseRoutes from './base'
import testRoutes from './test'

Vue.use(Router)

const routes = [...baseRoutes]

if (process.env.NODE_ENV !== 'production' && process.env.VUE_APP_ENABLE_TEST_ROUTE === 'true') {
  routes.push(...testRoutes)
}

export default new Router({ routes })
----

要求：

* 生产环境默认不注入测试路由。
* 如需线上灰度测试，必须通过明确环境变量开关控制。

== 6. 命名规范

* 页面文件建议：`XxxPage.vue` 或按历史约定保持一致（同项目内统一）。
* 业务域目录：使用清晰英文语义名，如 `auth`、`order`。
* 请求模型类型：`LoginRequest`。
* 响应模型类型：`LoginResponse`。
* 页面状态类型：`OrderListPageState`。

== 7. 反模式（禁止事项）

* `views/` 下平铺所有业务页面，不做业务域拆分。
* 将可复用业务组件误建为独立业务目录。
* 接口响应结构直接在页面中散落使用，不做统一模型定义。
* 测试页面路由直接混入正式路由并进入生产环境。

== 8. 落地与迁移建议

* 新需求严格按本规范新增目录与文件。
* 存量代码采用“增量迁移”，避免一次性重构风险。
* 代码评审必须检查：

1. 是否符合目录职责边界
2. 是否按规则定义 request/response/state 模型
3. test 路由是否已与生产路由隔离
