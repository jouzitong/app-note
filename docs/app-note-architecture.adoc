= 笔记系统架构设计方案
:toc: left
:toclevels: 4
:sectnums:

[perfer]
== 文档信息

- 项目名称：app-note
- 架构形态：Maven 多模块单体应用
- 后端技术栈：Spring Boot + Spring Data JPA + MySQL
- 安全能力：JWT 鉴权
- 当前目标：支撑“知识笔记 + 日语学习词卡 + 导入编排”这一条主业务链路

== 文档目标

本文档不是简单描述“有哪些模块”，而是用于指导后续开发落地，明确以下内容：

- 系统采用什么架构，以及为什么这样设计
- 物理模块与业务域模块如何划分
- 核心实体之间如何关联
- 各模块应承担什么职责，不应承担什么职责
- 后续新增功能时应该放在哪一层、依赖哪些能力

== 架构结论

=== 总体结论

本系统采用“*多模块单体 + 领域拆分*”架构：

- 部署上是单体应用，统一启动、统一数据库、统一鉴权
- 工程上拆成多个 Maven 模块，降低代码耦合
- 业务上按领域拆分，避免所有代码都堆在一个包里
- 数据上以“笔记节点 NoteNode”为核心主模型，其他业务能力围绕它扩展

该方案适合当前阶段，原因如下：

- 当前业务复杂度还不足以支撑微服务收益
- 笔记、词卡、例句、学习进度之间存在强关联，单体事务更简单
- 单体更利于快速迭代模型、接口和表结构
- 后续若某个领域膨胀，可以按模块拆分为独立服务

=== 一句话理解系统

系统的本质是：*用通用笔记节点组织知识结构，用领域模块承载不同类型的学习内容和学习行为。*

== 当前工程结构

基于当前代码，实际工程结构如下：

[source,text]
----
app-note
|- app          当前启动与应用编排混合模块
|- commons      公共配置、安全、基础设施
|- data-core    核心数据域（笔记节点、标签、内容）
|- server-word  日语词卡领域
----

建议后续演进为：

[source,text]
----
app-note
|- boot         启动模块
|- app-layer    应用层编排与聚合查询模块
|- commons      公共配置、安全、基础设施
|- data-core    核心数据域（笔记节点、标签、内容）
|- server-word  日语词卡领域
----

=== 物理模块职责

==== commons

公共基础设施模块，负责：

- Web 公共配置
- 跨域配置
- 鉴权拦截器与安全相关组件
- 后续可继续放：统一异常、返回体、审计、工具类、通用枚举

不建议放入：

- 具体业务实体
- 业务领域服务
- 领域专属 DTO / VO

==== data-core

核心数据域模块，是整个系统的基础模型层，负责：

- `NoteNode` 笔记节点主表
- `NoteNodeMeta` 节点扩展信息
- `NoteNodeContent` 节点内容
- `NoteTag` 标签体系
- 笔记树结构、路径、节点增删改查

它是整个系统的“知识结构骨架”，其他业务模块可以挂接在节点之上。

==== server-word

日语单词学习领域模块，负责：

- `WordCard` 单词卡
- `ExampleSentence` 例句
- `UserWordProgress` 用户单词掌握进度
- `Article` 文章与文章进度
- `WordCardNoteNodeRel` / `ArticleNoteNodeRel` 等领域与笔记节点的关联

它是一个明确的垂直业务域：依赖 `data-core`，但不反向影响 `data-core` 的通用性。

==== app（当前状态）

当前 `app` 模块同时承担了“启动”和“应用层编排”两类职责，负责：

- Spring Boot 启动
- 扫描所有模块实体与仓储
- 跨领域业务编排
- 导入服务
- 运维型接口或组合型接口

当前代码中已经体现出这一点：

- `NoteWordImportServiceImpl`：同时编排 `data-core` 与 `server-word`
- `NoteNodeOpsServiceImpl`：处理节点删除时的关联清理逻辑

从长期演进看，这类能力不建议继续和启动入口放在同一个模块中，建议后续拆分为 `boot + app-layer`。

==== boot（推荐新增）

`boot` 是纯启动模块，负责：

- Spring Boot 启动类
- 组件扫描与自动装配入口
- 各业务模块的装配与启动配置

`boot` 不负责：

- 业务 Controller
- 业务编排
- 聚合查询
- 导入、运维、调度等应用层流程

==== app-layer（推荐新增）

`app-layer` 是应用层模块，负责跨领域流程编排与视图组装，适合作为后续新增模块。

职责包括：

- 聚合查询
- 页面视图组装
- 导入流程编排
- 运维操作编排
- 跨领域写操作编排
- 启动后的初始化、补偿、调度型流程

典型适合放入 `app-layer` 的内容：

- 节点详情聚合查询
- 节点树 + 统计信息查询
- 节点下词卡掌握度、练习次数等聚合视图
- 导入“节点 + 词卡”等跨模块流程
- 删除节点时清理跨领域关联数据

`app-layer` 不负责：

- 定义核心领域实体真相
- 持有 `NoteNode`、`WordCard` 等主业务表归属
- 编写领域内部的底层规则
- 反向被领域模块依赖

一句话定义边界：

`app-layer` 负责“拿多个领域模块的数据与能力做编排和组装”，但不拥有领域本身。

== 推荐的逻辑分层

建议全系统统一采用以下分层：

[source,text]
----
Controller
  -> Application / Domain Service
      -> Repository
          -> Entity
----

更准确地说：

- `Controller`：接收请求、校验入参、返回结果
- `Application/编排服务`：跨领域组合流程，放在 `app-layer`
- `Domain Service`：领域内核心规则，放在各自领域模块
- `Repository`：数据访问
- `Entity`：持久化对象
- `DTO/VO/Convert`：输入输出模型与转换

=== 分层边界原则

- 跨模块流程编排放 `app-layer`
- 通用笔记树能力放 `data-core`
- 词卡、例句、学习进度放 `server-word`
- 公共技术组件放 `commons`
- `Controller` 不直接拼复杂业务流程
- `Repository` 不承载业务规则

== 业务域划分

从业务上看，建议系统按以下领域拆分。

=== 已落地或已有雏形的领域

- `auth`：用户与鉴权，当前主要由框架和 `commons` 支撑
- `note`：通用笔记节点体系，对应 `data-core`
- `word-learning`：日语词卡/例句/学习进度，对应 `server-word`
- `import`：导入编排，建议归入 `app-layer.importer`
- `ops`：运维操作与危险操作封装，建议归入 `app-layer.ops`
- `query`：聚合查询与页面视图组装，建议归入 `app-layer.query`

=== 规划中的领域

后续你提到的这些概念，建议不要一开始全都做成独立 Maven 模块，而是先作为业务域逐步落地：

- `practice`：题库、练习、错题
- `checkin`：打卡、连续学习记录
- `statistic`：学习统计、掌握度、趋势分析

建议顺序：

1. 先在单体内按包/领域扩展
2. 达到一定复杂度后再考虑升级为独立 Maven 模块
3. 只有在部署、团队协作、性能瓶颈明确时再考虑拆服务

== 核心设计思想

=== 笔记节点是顶层容器

`NoteNode` 是系统的统一知识容器，用来表达：

- 学习目录
- 章节结构
- 主题分组
- 卡片挂载位置
- 文章挂载位置

它负责统一知识结构容器，并通过 `noteType` 作为不同模板的入口；具体展示和领域行为再由各类型实现。

=== 内容与结构分离

当前 `data-core` 的设计已经体现出一个很重要的原则：

- `note_node`：结构信息
- `note_node_content`：正文内容
- `note_node_meta`：元信息

这样设计的好处：

- 节点列表查询时不必加载大段内容
- 结构变更与正文变更可以解耦
- 后续支持不同内容格式更方便

=== 领域对象通过关联表挂接到节点

例如：

- 单词卡通过 `word_card_note_node_rel` 关联到 `note_node`
- 文章通过 `article_note_node_rel` 关联到 `note_node`

这样可以避免把所有领域字段都塞进 `NoteNode`，保持核心模型稳定。

=== 行为记录独立建模

学习行为不建议混入静态内容实体，而应该独立建模，例如：

- `UserWordProgress`：记录用户对单词卡的掌握状态
- 后续可增加 `CheckinRecord`、`PracticeRecord`、`ArticleReadProgress`

这类表的本质是“用户行为事实表”。

== 核心实体模型

=== 核心实体关系图

[source,text]
----
NoteNode
  |- 1:1 -> NoteNodeMeta
  |- 1:1 -> NoteNodeContent
  |- parent_id -> NoteNode (树结构)
  |- N:N -> NoteTag (通过 NoteNodeMeta)
  |- 1:N <- WordCardNoteNodeRel -> N:1 WordCard
  |- 1:N <- ArticleNoteNodeRel -> N:1 Article

WordCard
  |- N:N -> ExampleSentence
  |- 1:N -> UserWordProgress

Article
  |- 1:N -> ArticleUserProgress
----

=== NoteNode

职责：

- 作为知识目录树的节点
- 描述父子结构、排序、标题、节点类型
- 为领域内容提供挂载点

当前关键字段含义：

- `noteKey`：业务唯一键，适合导入、同步、幂等处理
- `parentId`：树结构父节点
- `title`：节点标题
- `noteType`：节点模板或内容类型
- `sort`：同级排序
- `pathIds`：根到当前节点的路径 ID 列表，便于路径计算与树查询

设计建议：

- `pathIds` 属于冗余派生字段，必须由后端维护，不信任前端传值
- 节点删除、移动父节点后，需要同步重算子树路径

=== NoteNodeMeta

职责：

- 存放节点附加属性
- 承载轻量元数据，避免污染主表

当前包含：

- `icon`
- `subject`
- `tags`

后续可扩展但要克制，适合加：

- 封面
- 摘要
- 来源标记
- 展示样式

不建议加入复杂领域字段，例如词义、掌握度、练习次数。

=== NoteNodeContent

职责：

- 存放节点正文内容

当前字段：

- `content`，以 `TEXT` 存储 JSON 字符串

建议说明：

- 该字段本质上是“内容载荷”
- 适合存富文本 JSON、编辑器 Schema、导入原始结构
- 若后续需要全文检索，可考虑同步生成索引字段或搜索表

=== NoteTag

职责：

- 为节点或领域对象提供可复用标签能力

当前字段：

- `bizType`：区分标签适用的业务域
- `label`：标签名称
- `className`：展示样式类

设计价值：

- 通过 `bizType + label` 保证业务域内唯一
- 允许不同领域拥有同名标签但不同语义

=== WordCard

职责：

- 表示单词卡核心信息
- 挂接词义、同义词、相关词、例句、标签等内容

从现有实现看，`WordCard` 的设计理念是：

- 主表保存核心检索字段
- 复杂结构放入 `metaInfo`
- 多值关系如标签、例句单独建关联

这是合理的，因为词卡展示形态复杂，但查询入口相对稳定。

=== ExampleSentence

职责：

- 表示例句及其解析信息
- 支持多个单词卡复用同一例句

这意味着例句是独立知识资产，而不是某个单词卡的私有字段。

=== UserWordProgress

职责：

- 存储用户对单词卡的掌握进度
- 记录最近学习时间、状态、扩展行为信息

它与 `WordCard` 的关系体现了一个重要原则：

- 静态知识对象和用户行为对象必须分离

== 典型业务链路

=== 创建笔记节点

流程如下：

1. 前端提交节点基础信息与可选内容
2. `Controller` 调用 `INoteNodeDomainService`
3. 保存 `note_node`
4. 后端根据 `parentId` 计算 `pathIds`
5. 若有正文，则保存 `note_node_content`
6. 返回节点信息

当前实现中，这条链路已经基本具备。

=== 导入“节点 + 词卡”

当前 `NoteWordImportServiceImpl` 已体现出比较典型的应用编排流程：

1. 校验导入请求
2. 按 `noteKey` 导入或复用节点
3. 导入词卡
4. 将词卡关联到最后一个 `WORD_CARD` 类型节点
5. 汇总导入结果

这说明导入功能不是某个纯领域服务，而是典型的 `app-layer` 编排能力。

=== 删除节点及其关联数据

当前 `NoteNodeOpsServiceImpl` 的删除逻辑体现了“危险操作必须走专门服务”的原则：

1. 收集节点子树
2. 找到子树关联的词卡
3. 判断词卡是否仍被外部节点引用
4. 删除当前子树关系
5. 删除不再被引用的词卡
6. 删除不再被引用的例句
7. 最后删除节点子树

这个流程很重要，建议文档中长期保留，因为它是后续扩展 `practice`、`article` 等关联型能力时的参考模板。

== 推荐的包结构规范

后续开发建议在每个业务域内保持统一目录风格：

[source,text]
----
xxx
|- controller
|- service
|  |- impl
|- repository
|- entity
|  |- dto
|  |- meta
|  |- converter
|- dto
|- req
|- vo
|- convert
|- enums
|- utils
----

说明：

- `entity`：数据库实体
- `entity.meta`：JSON 扩展结构
- `repository`：JPA 仓储
- `service`：领域服务接口与实现
- `req/dto/vo`：输入、内部传输、输出模型
- `convert`：MapStruct 或手工转换

== 模块依赖规则

建议严格遵守以下依赖方向：

[source,text]
----
boot -> app-layer -> server-word -> data-core -> commons
boot -> app-layer -> data-core -> commons
boot -> commons
----

规则如下：

- `commons` 不依赖业务模块
- `data-core` 不依赖 `server-word`
- `server-word` 可以依赖 `data-core`
- `app-layer` 作为顶层应用编排模块，可以依赖所有领域模块
- `boot` 作为启动模块，可以依赖 `app-layer` 与各基础模块
- 领域模块不能反向依赖 `app-layer`

这样可以避免循环依赖，并保证核心模型稳定。

== 未来扩展建议

=== practice 练习域

建议拆成两类对象：

- 题目定义：题干、选项、答案、解析
- 行为记录：做题结果、耗时、错题标记

练习域建议也通过关联表挂到 `NoteNode`，而不是把练习字段塞进节点。

=== checkin 打卡域

建议核心对象：

- `CheckinRecord`
- `CheckinTarget`
- `CheckinStreakSnapshot`

这是典型行为域，建议和内容域分离。

=== statistic 统计域

建议后期实现，不建议一开始做复杂统计。

可先提供：

- 节点数统计
- 单词掌握数统计
- 最近学习趋势
- 连续学习天数

复杂聚合建议用单独查询服务，不要把统计逻辑散落在各个 Controller。

== 非功能设计建议

=== 事务边界

- 单领域写操作：由领域服务保证事务
- 跨领域写操作：由 `app-layer` 编排服务保证事务
- 删除、导入、批量绑定等操作必须显式定义事务边界

=== 幂等性

导入类场景建议统一依赖业务唯一键：

- `NoteNode.noteKey`
- `WordCard.cardCode`

后续其他领域也建议保留业务唯一键，便于同步和重复导入。

=== 删除策略

当前 `NoteNode` 使用逻辑删除基类，方向是合理的。

建议规则：

- 核心主对象优先逻辑删除
- 纯关系表直接物理删除
- 可复用的领域资源删除前先判断是否被其他对象引用

=== 查询性能

当前索引方向基本合理，后续可重点关注：

- `parent_id + sort`
- `note_key`
- `note_type`
- 关联表中的 `note_node_id`
- 领域主键的业务唯一字段，如 `card_code`

=== 内容存储策略

如果后续正文 JSON 变大，建议考虑：

- 只在详情页加载 `content`
- 列表页只查 `note_node + meta`
- 对热门查询字段做结构化冗余

== 开发落地建议

=== 第一阶段

优先完善核心骨架：

- 完整的 `NoteNode` 树能力
- 节点移动 / 删除 / 排序
- 标签与主题归类
- 词卡挂载与分页查询

=== 第二阶段

完善学习链路：

- 用户词卡学习进度
- 文章学习
- 节点详情聚合视图
- 导入流程稳定化

=== 第三阶段

扩展其他领域：

- `practice`
- `checkin`
- `statistic`

== 推荐的接口设计思路

建议接口按“核心能力”而不是“数据库表”暴露：

- 节点管理接口
- 节点树查询接口
- 词卡领域接口
- 导入接口
- 运维接口

不要把所有操作都做成通用 CRUD，否则后续复杂业务很快会失控。

== 最终建议

这套系统后续开发时，建议始终坚持以下原则：

- `NoteNode` 负责结构，不负责承载所有业务细节
- 各业务域通过关联表挂接到节点
- 行为记录独立建模
- 跨领域流程放 `app`
- 通用能力沉淀到 `data-core`
- 技术基础能力沉淀到 `commons`

如果按这个方向推进，后续即使增加文章学习、题库练习、打卡统计，也不会把系统做成一个无法维护的大泥球。
