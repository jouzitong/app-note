== 功能开发目标

基于当前系统定位，后续功能建设的核心方向不是“继续堆实体”，而是围绕以下三条主线推进：

- 把 `NoteNode` 打造成统一知识入口
- 把 `server-word` 打造成可闭环的学习领域
- 把 `app-layer` 打造成页面可消费的查询与编排入口

- 节点树路径能力

== 功能开发明细

=== F1：节点详情聚合查询

目标：

- 给前端一个稳定的“节点详情页”读取接口

建议放置：

- `app-layer.query`

建议输出模型：

- `NoteNodeDetailVO`

建议字段：

- 节点基本信息
- `meta`
- `content`
- 子节点摘要
- 挂载内容摘要
- 统计信息

建议接口：

- `GET /api/v1/nodes/{id}/detail`

实现思路：

- 从 `data-core` 读取节点主信息
- 按 `noteType` 判断是否需要加载领域内容
- 若为 `WORD_CARD` 节点，加载词卡统计
- 若为 `ARTICLE` 节点，加载文章摘要

验收标准：

- 一个接口可满足节点详情页基础展示
- 前端无需自行拼多次请求

=== F2：节点树查询

目标：

- 提供目录树、知识树、章节树的统一查询能力

建议放置：

- `app-layer.query`

建议输出模型：

- `NoteNodeTreeVO`

建议字段：

- `id`
- `title`
- `noteType`
- `sort`
- `path`
- `children`
- 可选 `stats`

建议接口：

- `GET /api/v1/nodes/tree`
- `GET /api/v1/nodes/{id}/children`

实现重点：

- 支持按根节点查询子树
- 支持懒加载 children
- 支持节点基础统计摘要

验收标准：

- 前端目录树不需要自己递归拼装多个接口

=== F3：节点统计查询

目标：

- 给节点展示掌握度、练习次数、最近学习时间等指标

建议放置：

- `app-layer.query`

建议输出模型：

- `NoteNodeStatsVO`

建议字段：

- `type`
- `totalCount`
- `masteredCount`
- `practiceCount`
- `masteryRate`
- `lastLearnAt`

实现策略：

第一阶段先做实时聚合：

- 从 `WordCardNoteNodeRel`
- `UserWordProgress`
- 后续 `PracticeRecord`

第二阶段视性能决定是否引入快照表：

- `note_node_stat`

验收标准：

- 节点详情、节点树可挂统计摘要

=== F4：词卡分页与详情优化

目标：

- 让 `server-word` 能支撑完整词卡学习页

现有基础：

- 词卡分页
- 词卡详情
- 掌握确认

建议补充：

- 下一个未掌握词卡查询
- 指定节点下学习进度摘要
- 最近复习列表
- 词卡搜索

建议接口：

- `GET /api/v1/word-cards/next`
- `GET /api/v1/word-cards/progress-summary`
- `GET /api/v1/word-cards/search`

验收标准：

- 词卡学习页面可仅依赖少量接口完成主流程

=== F5：词卡学习闭环

目标：

- 形成“看词卡 -> 确认掌握 -> 记录进度 -> 展示统计”的闭环

需要补齐：

- 词卡学习状态枚举梳理
- 掌握 / 未掌握 / 待复习 状态流转
- 复习时间记录
- 进度汇总查询

建议拆分任务：

- F5-1：规范 `UserWordProgressStatus`
- F5-2：增加未掌握确认接口
- F5-3：增加复习时间写入
- F5-4：增加用户维度进度汇总

=== F6：文章学习基础能力

目标：

- 让 `Article` 从“存在实体”升级为“可使用能力”

建议优先补齐：

- 文章分页
- 文章详情
- 文章与节点关联查询
- 用户文章进度

建议接口：

- `GET /api/v1/articles`
- `GET /api/v1/articles/{id}`
- `POST /api/v1/articles/{id}/progress`

验收标准：

- 文章能像词卡一样被节点挂载和浏览

=== F7：练习域设计与实现

目标：

- 建立题目练习与错题复盘能力

建议拆为两个部分：

- 静态题目定义
- 用户练习记录

建议实体：

- `Question`
- `QuestionOption`
- `QuestionNoteNodeRel`
- `PracticeRecord`

建议第一期能力：

- 题目录入
- 题目挂接节点
- 提交答题结果
- 错题标记
- 按节点统计练习次数

建议模块归属：

- 新建领域模块或先在单体内按包扩展

=== F8：打卡域设计与实现

目标：

- 支撑连续学习与日常学习记录

建议实体：

- `CheckinRecord`
- `CheckinTarget`
- `CheckinSnapshot`

建议第一期能力：

- 用户每日打卡
- 查询连续天数
- 查询最近打卡记录

统计展示位置：

- `app-layer.query`

=== F9：统计域

目标：

- 给首页、仪表盘、学习概览页提供统一统计能力

第一期指标建议：

- 节点总数
- 词卡总数
- 已掌握词卡数
- 最近 7 天学习次数
- 连续学习天数

实现建议：

- 先实时聚合
- 热点指标后续快照化

=== F10：搜索能力

目标：

- 提供统一的知识检索入口

建议范围：

- 节点标题搜索
- 标签搜索
- 词卡搜索
- 文章搜索

第一阶段实现：

- 基于数据库字段查询

后续可扩展：

- 全文检索
- 条件组合筛选

== 接口设计原则

- 以页面能力为中心设计接口
- 不要默认按表暴露 CRUD
- 聚合查询统一通过 `app-layer.query`
- 写操作按命令职责划分

== 验收标准

满足以下条件时，可认为后续功能开发文档进入可执行状态：

- 每项功能都能找到模块归属
- 每项功能都有明确输入输出模型
- 第一阶段功能可以独立开工
- 后续扩展域不会再与现有架构冲突
